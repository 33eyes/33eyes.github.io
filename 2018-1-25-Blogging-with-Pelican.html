Blogging with Pelican

  • Thu 25 January 2018
  • Tutorials
  • #blogging, #pelican, #git, #Windows, #Python

Static site generators are a great option for blogging. They are faster, cheaper and more secure than traditional CMS blogging platforms like Wordpress. In addition, static site generators also offer straight-forward version control of the entire site, making it easy to restore earlier versions of the site if needed. There are lots of blog posts out there extolling the benefits of static site generators in great detail, so I will skip repeating them here, and list a few in the reference section below.

Why Pelican?

So, static site generators are good stuff. There are hundreds of static site generators out there, and developers keep coming up with new ones. Currently, the most popular one is Jekyll. It's simple, Ruby-based, very easy to publish on GitHub, and has a large and active community. Many other static site generators are pretty great as well, and which one you choose depends your preferences and goals. Some static site generators are better suited for blogs, others for documentation, some are more customizable than others, some are heavy on JavaScript, some can be hosted on GitHub or Dropbox, etc. And of course, they are written in a variety of languages, including Ruby, Go, Python and JavaScript. After briefly trying out Jekyll, I've decided to go with a different static site generator for my blog: Pelican.

Here are my very subjective reasons for choosing Pelican over Jekyll:

  • Pelican is Python-based, and Jekyll is Ruby-based. While I'm familiar with both languages, I use Python for data science projects, and this blog is about data science, so I thought it would be easier to stick with Python.

  • Pelican seems to me more customizable than Jekyll. Pelican's themes are easy to switch, because they live in a separate folder and you simply point to the theme's folder in your config file to appy it to your website. Pelican also offers plugins.

  • Pelican is more Windows-friendly. Local development with Jekyll on my Windows 10 machine did not work for me. Jekyll is so simple that this is not a big problem, because you can easily use GitHub for all your publishing needs with Jekyll, or you can clone your Jekyll blog repo to cloud9 (or some other virtual linux box) and develop it locally there. Local development with Pelican on Windows did work for me, although with some adjustments. While this is not crucial, I consider it a plus.

Setup

Installation

First, make sure that you have git, Python and pip installed. If using Windows, I recommend using Git Bash for running the unix terminal commands below (it's included with the git distribution for Windows).

Install pelican and markdown

Run the following command in your terminal: 

pip install pelican markdown

Create a project for your blog

Create a directory for your new project, cd into it in your terminal, and run the following command:

pelican-quickstart

This command will create a skeleton of your pelican project. It will ask you some questions when you run it. You can hit Enter to pick the default answers where available. All you really need to enter is the website title and author. The settings this script asks for can be adjusted later in the project's config file.

Create your first blog post

You need to create some content before you can generate your site with pelican. Let's make a quick "hello world!" blog post to get started. In a text editor of your choice, enter something like this:

Title: Hello, world!
Date: 2018-01-25 12:50
Category: Test

Hello, world! This a test blog post.

Save this text file as hello-world.md in the content folder of your pelican project.

By default, the content folder is where all of your Markdown content files need to be so that pelican can process them when generating the static site. Chronological content (i.e. blog posts) go into the main content folder, and if you'd like to have non-chronological content, like an About page, you'll need to create a pages subfolder inside the content folder.

Generate your static site

From your project's folder, run the following command:

pelican content

This command processes your content inside the content folder and saves the static webpage versions of your content in the output folder of your project.

Run your site locally

There are a couple of options for how to do this in pelican, this is just one of them.

From your project's folder, run the following command:

make devserver

This command starts your local development server. You can view your site on localhost:8000 .

While this command runs in the background and you can still use the same terminal window, it outputs server status messages to that window, which can get confusing if you're trying to run other commands there. So it's easier to just leave this terminal for the server, and do other work in a new terminal window.

The command for stopping the development server is:

make stopserver

Since the make devserver command runs in the background, you can't just Ctrl+C out of it.

Customization

Settings

Pelican has two settings files in the main project folder: pelicanconf.py and publishconf.py. When you run the local development server with make devserver, pelican uses just the pelicanconf.py settings file. This is the file you will use most of the time to configure your site. Many of the settings here are used by themes.

When you are ready to publish your website, you can run make publish, and pelican will generate the static site using the settings you've specified in pelicanconf.py and the settings in publishconf.py. The publishconf.py settings file imports pelicanconf.py at the start of the file, and whatever settings you specify in publishconf.py after the import will overwrite the pelicanconf.py. One example of using that is setting the SITEURL prefix. You'll want SITEURL = '' in your pelicanconf.py for local development, and SITEURL = 'http://yourusername.github.io' in your publishconf.py for publishing your site to GitHub pages.

Themes

Pelican makes it very easy to change themes. All you need is to have a folder with your pelican theme(s) in your project folder, and then just point to the theme's folder in your pelicanconf.py file: 

THEME = "./pelican-themes/my-custom-theme"

You can clone or download pelican themes from the pelican theme repository, or create your own.

Hosting on GitHub Pages

  1. Create an empty repo on GitHub, and call it yourusername.github.io, subbing in your GitHub username. Make sure its visibility is set to public.
  2. Use git to push the contents of your output folder onto the master branch of your GitHub repo.
    There are a couple ways to do this:
    • ghp-import is a great plugin if you are on a Mac or unix-based OS, but not so great if you have Windows. There is a Windows workaround for this (see the note), but it did not work on my machine. All ghp-import does is create a branch called gh-pages and put only the contents of your output folder on it. Then, every time you run it, it refreshes those contents. To install ghp-import on a Mac/unix-ish OS, run: pip install ghp-import. (For Windows, see if the workaround linked above works for you. If not, the alternative way below should work.) Once that's installed, you can run ghp-import output to commit the contents of your outout folder onto the gh-pages git branch. After that, push your gh-pages local branch onto your remote GitHub Pages repo master branch by running the following: git push git@github.com:yourusername/yourusername.github.io.git gh-pages:master .
    • Another way to do this is to just copy your output folder somewhere outside your pelican project folder, initialize it as its own git repository, commit everything to the local master branch, and push that local master branch to the remote GitHub Pages repo master branch. This does mean that you'll have to copy over your output folder every time you want to publish to GitHub, but it is just one cp command instead of the ghp-import, and then remember to push to GitHub from that external output folder. It's pretty easy to write a script to do this as well, and this approach works fine on Windows.

Note: GitHub might take a while to generate your website when you push it for the first time.

My blogging workflow

  1. cd into the pelican project folder, and start up the local development server with make devserver .

  2. Write some content / make some customizations

    • the blog posts should go into the content folder
    • static pages should go into content/pages subfolder
    • blog/theme settings can be adjusted in pelicanconf.py config file
    • themes can be added or modified in pelican-themes folder

  3. Review the local changes

    • open localhost:8000 in browser and review the site
    • once happy with the local changes, shut down the development server with make stopserver .

  4. Commit changes to my blog development repo

    • Save blog development changes to a separate repo.

  5. Generate production output

    • run make publish

  6. Push the new static output to my GitHub Pages repo

    • copy the contents of output folder (either to the gh-pages branch via ghp-import, or to an external output folder with its own repo)
    • push the new static output to the master branch of the GitHub Pages repo.
